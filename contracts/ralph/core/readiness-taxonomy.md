# Ralph Readiness And Label Taxonomy

## Purpose

Define a deterministic label model where automation eligibility is based on readiness, while provenance remains informational.

## Label Dimensions

1. Automation eligibility
   - `Ralph`: issue is in the Ralph execution domain.
2. Implementation readiness
   - `PRD Ready`: issue has sufficient implementation detail for unattended execution.
3. Human intervention state
   - `Human Required`: automation is paused pending human input.
4. Provenance
   - `Agent Generated`: indicates the issue was generated by an agent. This is metadata, not a readiness gate.

## Eligibility Rule

An issue is eligible for unattended Ralph execution when all conditions are true:

- has label `Ralph`
- has label `PRD Ready`
- does not have label `Human Required`

`Agent Generated` must never be used as an eligibility requirement.

## Default Operating Policy

- 95% default path: issues run through the agent-managed flow with readiness semantics.
- 5% exception path: human-authored and/or human-executed work is explicitly marked and can bypass automation until made `PRD Ready`.

## PRD-Ready Structure Requirement

For `PRD Ready`, an issue description should contain:

- clear problem framing
- implementation notes with concrete file/component targets
- explicit dependencies
- acceptance criteria
- validation commands or expected checks

Reference template: `templates/linear/prd-ready-issue.md`.

## Metadata Expectations

Readiness and metadata are separate concerns:

- Readiness labels determine automation eligibility.
- Metadata quality determines planning/scheduling quality.

For automation-targeted issues, metadata should be rubric-derived from `issue-metadata-rubric.md` and include:

- deterministic `priority`
- deterministic `estimate`
- `estimatedTokens` (recommended)
- concise `Metadata Rationale`

## Backlog Migration Policy

For existing issues in a project migration:

1. Keep provenance as-is (`Agent Generated` only when true).
2. Add `Ralph` only to issues intended for the automation domain.
3. Add `PRD Ready` only after structure/readiness audit passes.
4. Add `Human Required` for blocked or ambiguous items needing human decisions.
5. Remove `PRD Ready` when unresolved ambiguity or failed retries makes the issue unsafe for unattended execution.

## Command Contract Alignment

- `audit-project` validates readiness semantics.
- `populate-project` applies readiness labels to generated automation-targeted issues.
- `ralph-run` selects from readiness-eligible issues only.
- Claim lifecycle details are defined in `claim-protocol.md` and must be applied before execution begins.
